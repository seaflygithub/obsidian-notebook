## 🧱 模块2：数据发生器（你自己写）

在 `clk_fast` 下：这就是“数据源”
```verilog
reg [31:0] data_cnt;

always @(posedge clk_fast) begin
    if (!rst_n)
        data_cnt <= 0;
    else
        data_cnt <= data_cnt + 1;
end
```

下面是实现的同样的功能，顺便增加握手功能，用的标准的 systemverilog 编写的(data_source.sv):
```verilog
`timescale 1ns / 1ps

module data_source (
    input  logic        clk_fast,
    input  logic        rst_n,
    input  logic        enable,
    input  logic        ready,

    output logic [31:0] data_out,
    output logic        valid
);

logic [31:0] data_cnt;
logic        handshake;

// 抽象出握手信号,方便后续修改扩展
assign handshake = valid && ready;

// valid
always_ff @(posedge clk_fast or negedge rst_n) begin
    if (!rst_n)
        valid <= 1'b0;
    else
        valid <= enable;
end

// data path
always_ff @(posedge clk_fast or negedge rst_n) begin
    if (!rst_n) begin
        data_cnt <= 32'd0;
        data_out <= 32'd0;
    end else if (handshake) begin
        data_cnt <= data_cnt + 1;
        data_out <= data_cnt;
    end
end

endmodule
```





下面是模块的单独仿真测试，完整激励文件(data_source_tb.sv)内容如下：

```verilog
`timescale 1ns / 1ps

module data_source_tb;

//--------------------------------------------------
// 信号定义
//--------------------------------------------------
logic clk_fast;
logic rst_n;
logic enable;
logic ready;

logic [31:0] data_out;
logic        valid;

//--------------------------------------------------
// DUT 实例化
//--------------------------------------------------
data_source dut (
    .clk_fast (clk_fast),
    .rst_n    (rst_n),
    .enable   (enable),
    .ready    (ready),
    .data_out (data_out),
    .valid    (valid)
);

//--------------------------------------------------
// 时钟：100MHz (10ns周期)
//--------------------------------------------------
initial clk_fast = 0;
always #5 clk_fast = ~clk_fast;

//--------------------------------------------------
// 激励
//--------------------------------------------------
initial begin
    // 初始值
    rst_n  = 0;
    enable = 0;
    ready  = 0;

    // 复位一段时间
    #20;
    rst_n = 1;

    // 打开数据源，但下游不ready
    #10;
    enable = 1;
    ready  = 0;

    // 观察：此时 valid=1，但 data_out 不变化
    #50;

    // 下游开始接收
    ready = 1;

    // 观察：data_out 开始递增
    #100;

    // 制造背压（ready=0）
    ready = 0;

    // 观察：data_out 应保持不变
    #50;

    // 再恢复
    ready = 1;

    #100;

    // 关闭数据源
    enable = 0;

    #50;

    $finish;
end

//--------------------------------------------------
// 监视输出（打印波形信息）
//--------------------------------------------------
initial begin
    $display("time\tvalid ready data_out");
    $monitor("%0t\t%b     %b     %0d", 
             $time, valid, ready, data_out);
end

//--------------------------------------------------
// 可选：自动检查（简单断言）
//--------------------------------------------------
logic [31:0] last_data;

always @(posedge clk_fast) begin
    if (!rst_n)
        last_data <= 0;
    else begin
        // 如果没有握手成功，数据必须保持
        if (!(valid && ready)) begin
            if (data_out != last_data) begin
                $error("ERROR: data changed without handshake!");
            end
        end
        last_data <= data_out;
    end
end

endmodule
```

下面必须要知道的基础知识：

```verilog
// 组合逻辑-特点: 没有时钟、输入一变输出立刻变(组合路径延迟除外)
assign handshake = valid && ready;
```

---

```verilog
// 多条逻辑有依赖关系（标准工程写法）
always_comb begin
    logic t1;
    logic t2;

    t1 = b & c;
    a  = t1;

    t2 = a | e;
    d  = t2;

    f  = d ^ g;
end

/*
b & c → t1 → a
             ↓
           a | e → t2 → d
                        ↓
                      d ^ g → f
**/


```

这里为什么插入中间器件t1,t2：
- 从人类角度，提高可读性、方便调试、方便加pipeline；
- 方便插入寄存器（架构扩展）；
- 方便调试抓信号，你可以直接 probe t1 t2；
- 人为给组合逻辑绘制分段电路图，让设计可读、可验证、可扩展。


---

```verilog
// 如果没有else兜底, 那么a一定要事先赋初值
always_comb begin
    a = 0;
    if (b)
        a = c;
end
```

综合器做了这个推理：
“**你没有告诉我 b=0 时 a 应该是什么，那我只能假设：a 要保持不变**”

👉 而“保持不变”在硬件里只能用：
> ✔ 反馈回路 + 存储结构 = latch

“**如果你不说新值，那我就帮你记住旧值**”
“每一刻你都必须告诉我 a 是多少”


---


❗**非阻塞赋值 `<=` 是“同时采样旧值、同时在时钟沿更新”**  
而不是“按代码顺序一行一行执行”
```verilog
if (handshake) begin
	data_cnt <= data_cnt + 1;   // 用旧值(data_cnt=1) → 计算得到 2
	data_out <= data_cnt;       // 用旧值(data_cnt=1)
end

/*
clk:       ↑     ↑     ↑
data_cnt:  0     1     2
data_out:  0     0     1
**/
```

---

**一个非常关键的直觉**（你要建立）

以后你看到：always_ff @(posedge clk)
脑子里要自动变成：
**👉 “这里有一排寄存器，在时钟沿同时更新”**

看到：always_comb
脑子里变成：
**👉 “这里是一坨组合逻辑，一直在计算”**

<font color=blue>在仿真过程中，当 enable 刚好在 clk上升沿时拉低，那么valid为什么也会跟着拉低？也就是说valid为什么会判断 enable 为低？在时钟边沿 enable 不是处于半高半低状态么？原来仿真的是变化之后的结果了，那真实硬件并不一定这样，真实硬件确实可能会刚好在时钟上升沿发生信号变化，这时候的信号处在亚稳态。在工程中，如何避免这种亚稳态？
</font>

❗**亚稳态不可避免，但可以通过“同步器 + 正确架构”把风险降到几乎为零**

1bit 的信号，
❗两级同步器 = 工程里处理单bit跨时钟/异步信号的标准方案
```verilog
logic en_ff1, en_ff2;
always_ff @(posedge clk_fast) begin
    en_ff1 <= enable;
    en_ff2 <= en_ff1;
end
valid <= en_ff2;
```

❗注意：**所有信号都用的同一个时钟(clk_fast)，这种不需要同步器！**


如果不是单bit，而是多bit：
**1️⃣ FIFO（最常用）**
写时钟域 → FIFO → 读时钟域
👉 自动处理同步问题
**2️⃣ 握手机制（valid/ready）**
valid → 同步 → 对方
ready → 同步 → 返回

❗**同步器只解决“时钟域问题”，和“是不是别的模块”没有关系**


---

<font color=blue>那我们常见的各种ip核 or 模块，其参数信号 read, valid，这种，是不是就等同于它已经使用了同步器？我们使用时按照它的握手信号访问即可？
</font>

```txt
producer ──valid/ready── consumer
      ↑                    ↑
   clk_fast           clk_fast
```
👉 ✔ 同一个时钟域

valid / ready 只是协议里的流控概念。

<font color=blue>帮我举个例子，如果是跨时钟域，同一个模块内部，两个 always_ff，其中一个是 clk1，另一个是clk2，分别帮我给单bit信号加同步器，以及多bit的数据加同步器。
</font>

很好，这个问题已经进入 **CDC（跨时钟域）设计的核心实战**了。我给你两个**最小但工程可用的例子**：
- ✔ 单 bit 信号（用两级同步器）
- ✔ 多 bit 数据（用握手/打包同步，避免“逐位同步”的错误）

🧠 一、单 bit 跨时钟（正确做法：**两级同步器**）
🎯 场景
- clk1 域产生一个控制信号 `flag_clk1`
- clk2 域要用它

```verilog
module cdc_single_bit (
    input  logic clk1,
    input  logic clk2,
    input  logic rst_n,

    input  logic flag_clk1,   // clk1 域
    output logic flag_clk2    // clk2 域（同步后）
);

//----------------------------------
// clk2 域同步器（两级FF）
//----------------------------------
logic sync_ff1, sync_ff2;

always_ff @(posedge clk2 or negedge rst_n) begin
    if (!rst_n) begin
        sync_ff1 <= 0;
        sync_ff2 <= 0;
    end else begin
        sync_ff1 <= flag_clk1;   // 第一级（可能亚稳）
        sync_ff2 <= sync_ff1;    // 第二级（稳定输出）
    end
end

assign flag_clk2 = sync_ff2;

endmodule
```

**🧠 原理图（一定要有这个直觉）**

```txt
clk1: flag ───────────────┐
                          ▼
clk2:         [FF1] → [FF2] → flag_clk2
                 ↑       ↑
              可能亚稳   已稳定
```

✅ 三、多 bit 正确做法（握手 + 打包）

🎯 思路
> ❗**不要同步“数据”，只同步“控制信号”**

📦 方案：valid + 同步 + 数据锁存

工程级更安全方案（推荐）

如果你要真正“工业级可靠”，用：
✔ 方案1：握手（req/ack） —— 即双向确认
**✔ 方案2：Async FIFO（最推荐）**

```
clk1 → FIFO → clk2
```

## 🧱 模块3：FIFO（重点IP）

IP选择：
- FIFO Generator
配置：
- 接口类型: Native、AXI Stream，选择Native（因为Native更接近底层，需要自己管理empty和full信号）
- FIFO类型：**Independent Clocks（异步）**
- 宽度：32bit
- 深度：1024
- 打开：
    - `full`
    - `empty`
    - `wr_data_count`（可选，调试很好用）

---

**写端（clk_fast）**
```verilog
assign wr_en = !full;

always @(posedge clk_fast) begin
    if (wr_en)
        din <= data_cnt;
end
```

---

**读端（clk_slow）**
```verilog
assign rd_en = !empty;

always @(posedge clk_slow) begin
    if (rd_en)
        fifo_out <= dout;
end
```

---

👉 你可以观察现象：
- 写快（100MHz）读慢（25MHz）
- FIFO会越来越满 → full亮

---

## 🧱 模块4：RAM（Block Memory）

IP：
- Block Memory Generator
配置：
- True Dual Port RAM（推荐）
- 宽度：32bit
- 深度：1024

---

**写入（来自FIFO）**

```verilog
always @(posedge clk_slow) begin
    if (rd_en) begin
        ram[wr_addr] <= fifo_out;
        wr_addr <= wr_addr + 1;
    end
end
```

---

**读取（可以慢速读 or 按键触发）**

```verilog
always @(posedge clk_slow) begin
    read_data <= ram[rd_addr];
    rd_addr <= rd_addr + 1;
end
```

---

👉 关键理解：
- RAM 是“存储历史数据”
- FIFO 是“流式缓存”

# 🔍 实验现象（必须观察）

### 🧪 实验1：写快读慢

- FIFO → 满
- RAM写入连续数据

---

### 🧪 实验2：读快写慢（改频率）

- FIFO → 空
- 数据断断续续

---

### 🧪 实验3：停止写入

- FIFO逐渐被读空
