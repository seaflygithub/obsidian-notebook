
寄存器富集现象，一句话描述：
👉 所以对应现象是：
> 🔥 **“高性能资源被过量占用 → 请求溢出到低性能路径”**


我们先从一个典型的生产事故说起。某电商平台在双11预热期间，订单创建接口突然出现大量超时，监控显示数据库连接池耗尽，但慢查询日志里其实没有明显问题。后来发现，问题出在事务回滚时，大量未提交的事务在数据库层面被反复执行，而根本原因是：系统在处理请求时，把大量业务状态、用户上下文、会话信息等 "寄存" 到了数据库事务中，而不是本地缓存或内存结构里。这种做法，本质上就是寄存器富集的典型滥用。

寄存器富集，字面意思是 "**把大量数据塞进寄存器**" —— 这里的寄存器不是指计算机里的寄存器，而是比喻系统中那些临时存储、快速访问但容易被污染或耗尽的资源。正常情况下，这些 "寄存器" 是高效的，但一旦被塞进太多无关或冗余数据，就会导致资源竞争、上下文切换频繁、GC压力激增，甚至引发连锁崩溃，这现象就是寄存器富集。

**为什么系统要这么做？**
因为早期微服务架构中，服务之间通信依赖大量外部系统：消息队列、配置中心、鉴权服务、日志上报、监控采集。。。每个调用都可能携带大量上下文信息，比如用户ID、请求来源、时间戳、Trace ID、服务版本、环境变量等。为了保证链路可追踪、审计可查、故障可定位，系统必须把这些信息寄存到调用链中。但问题来了：这些信息很多是重复的、临时的、甚至无用的。比如同一个 Trace ID 在多个服务间传递，但每个服务都重新计算一次，内存中不断堆积未清理的局部变量，最终导致内存泄漏或频繁GC。

更深层的问题是，寄存器富集本质上是一种 "**为安全而牺牲性能**" 的设计。系统为了保证在异常情况下还能恢复状态、记录操作轨迹、支持回滚，不得不把大量数据寄存在临时结构中。这就像你在写代码时，为了防止某个函数出错，把所有输入参数、中间结果、调用栈都压入一个全局栈中，看起来很安全，但一旦这个栈溢出，整个系统就瘫痪了。

</font>

👉 关键点：
> ❗问题不是“并发”，而是“并发 + 某个薄弱点”

比如：
- 高并发 + 缓存同时过期
- 高并发 + 锁粒度错误
- 高并发 + IO 变慢（磁盘/网络抖动）

正确姿势：构造“**组合极端场景**”
✅ 正确方式：
如果：  
- 并发上升  
- + 缓存失效  
- + DB 延迟变高  
会发生什么？
👉 这才接近真实事故

🔥 **系统稳定性公式**
稳定性 = 限流能力 + 隔离能力 + 降级能力

👉 架构师不是“假设流量很大的人”，而是：
能系统性地预见“**哪些组合条件会让系统崩溃**”，并提前切断这些路径的人
