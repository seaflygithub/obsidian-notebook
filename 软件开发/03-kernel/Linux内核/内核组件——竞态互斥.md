# smp_wmb与wmb选择
【回答】smcdef建议，如果内存操作的顺序的观察者全是CPU，那么请使用smp_[w,r]mb()。如果内存操作顺序涉及CPU和硬件设备（例如网络设备），请使用[w,r]mb()。另外，[w,r]mb()比smp_[w,r]mb()的要求更严格。因此性能方面也是[w,r]mb()比smp_[w,r]mb()差。如果我们不写驱动的话，其实很少和设备打交道。因此一般都是选择smp_[w,r]mb()。

# wmb和mb选择
【回答】rmb,wmb,mb这三类，我们为了可移植性，一般情况下推荐使用mb指令，理由如下:
```cpp
//linux-4.0/arch/arm64/include/asm/barrier.h
#define mb()        dsb(sy)
#define rmb()       dsb(ld)
#define wmb()       dsb(st)

#define dma_rmb()   dmb(oshld)
#define dma_wmb()   dmb(oshst)

#ifndef CONFIG_SMP
#define smp_mb()    barrier()
#define smp_rmb()   barrier()
#define smp_wmb()   barrier()
#else
#define smp_mb()    dmb(ish)
#define smp_rmb()   dmb(ishld)
#define smp_wmb()   dmb(ishst)
#endif //CONFIG_SMP

//ISHLD     Load-Load,Load-Store
//ISHST     Store-Store (只保证前后store的顺序,其他指令不管)
//ISH       Any-Any (所有指令LD,ST都不能跨过屏障插队执行)
```

## dmb和dsb以及isb
如上代码所示，比如smp_mb()，这个以smp_开头的都使用了最宽松的dmb，即屏障只挡住闸门前后的存储指令，其他指令放行，也就是确保前面的存储类指令执行完了，才放行后面的存储类指令进入流水线。而没有smp_前缀的mb()，则使用了中等严格的dsb，即屏障保证前面的存储类指令执行完了，才放行后面的所有指令进入流水线。isb是最严格的屏障，必须保证前面所有指令都执行完了，才放行后面的所有指令进入流水线。

想象这样一个场景，流水线门外，有4个通道，每个通道都能进人，每个人都是一条指令，不同人代表不同指令。这些指令按顺序进入流水线，守门员负责把关。当守门员看到某条通道上dmb指令过来时，守门员会只挡住dmb身后所有的存储类指令，其他类别的指令放行进入流水线，等流水线内的存储类指令执行完了，才放行dmb身后的存储类指令。当守门员看到某条通道上走过来dsb时，守门员会挡住dsb身后所有类别指令，等流水线内的存储类执行完了，才放行dsb身后的所有指令。当守门员看到某条通道上走过来一个isb时，这时守门员会挡住isb身后所有指令，等流水线内所有指令都执行完了，才放行isb身后所有指令。


# 深度思考（乱序执行与内存屏障）

```cpp
a = 1;
b = 2;
c = 3;
d = 4;
e = 5;
f = 6;
if (f == 6)   //<---乱序情况下f=6有可能优先于c=3执行，导致f == 6时c不一定为3
	data = c;
```

解决办法就是要保证if语句访问c时，c已经被赋值了。一种写法是在c赋值后紧接着放入屏障以确保c赋值了，另一种写法是在if语句需要访问c之前放入屏障以确保c已经被赋值好了。如果两处都加屏障的话，会影响CPU执行指令的效率性能。
```cpp
a = 1;
b = 2;
c = 3;
smp_mb(); //<---确保在if语句访问时已经给c赋值了
d = 4;
e = 5;
f = 6;
//smp_mb(); //<---确保在if语句访问时已经给c赋值了
if (f == 6)
	data = c;
```




# 内存一致性模型PSO：WW乱序

【回答】在TSO模型中，我们说过store buffer会按照FIFO的顺序将数据写入L1 Cache。

```cpp
x = 10;
y = 1;
```

【回答】我们知道按照TSO模型的执行思路，首先x=10的值会写入store buffer，同样y=1的值也会写入store buffer。然后按照FIFO的次序写入L1 Cache。但是**store buffer又优化了**，不再以FIFO方的次序写入L1，也就是说支持**插队**了，这在一定程度又可以提升性能。此时，store buffer不再以FIFO的次序写入Cache。这种硬件优化会对程序的运行又会造成什么影响呢？
```cpp
int flag, data;
static void write_data_cpu000000(int value)
{
  data = value; //store
  flag = 1;     //store
  //对于CPU来说是不知道data和flag之间是有依赖关系的，
  //CPU认为data和flag的值谁先写入Cache都无所谓。
  //所以做出了这种优化。
}
static void rread_data_cpu111111(void)
{
  while (flag == 0);
  return data;
}
```

【回答】我们知道TSO模型可以看成store-load操作可以乱序成load-store。现在，我们提出了一种新的内存模型，部分存储定序(Part Store Order)。**PSO在TSO的基础上继续允许store-store乱序**。还是那句话，CPU0看自己的内存操作是顺序的，只是CPU1看CPU0内存操作可能是乱序的。现在我们可以忘记store buffer的存在，从内存模型的角度考虑这个程序。data和flag操作都是store，所以按照PSO模型来说，其他CPU是可以看到乱序操作的结果。这样就很容易理解了。尽量不去思考硬件为什么会这样，而是认为这是PSO模型理所应当存在的现象。

【回答】Linux内核中提供了 smp_wmb() 宏对不同架构的指令进行封装， smp_wmb() 的作用是阻止它后面的写操作不会乱序到宏前面的写操作指令前执行。 smp_wmb() 充当的就是一个 store-store 屏障。但是这个屏障只针对store操作，对load操作不影响。
```txt
store---------+
              |
              v
-------store-store barrier--------
              ^
              |
store---------+
```

```cpp
static void write_data_cpu0(int value)
{
  data = value;
  smp_wmb();
  flag = 1;
  //smp_wmb()可以保证CPU1看到flag的值为1时，data的值一定是最新的值。
}
```





# 内存一致性模型RMO

【回答】硬件的优化总是追求极致的。硬件工程师在不满足于PSO的基础上提出了更弱的内存模型RMO。按照常理来说，理应有一种内存模型允许4种操作都可以乱序。这就是Relaxed Memory Order，简称RMO。例如aarch64就是典型的RMO模型。

【回答】继续前面的代码，当我们在write_data_cpu0()加入smp_wmb()后还不足以解决问题。我还需要在读取测加入smp_rmb()屏障。因此完整修改如下。

```cpp
static int flag, data;
static void write_data_cpu0(int value)
{
  data = value;
  smp_wmb();
  flag = 1;
}
static void read_data_cpu1(void)
{
  while (flag == 0);
  smp_rmb();
  return data;
}
```

【回答】在之前的文章中，我们已经见识了 smp_mb() 和 smp_wmb() 的作用，这里引入一个新的读屏障 smp_rmb() 。 smp_rmb() 保证屏障后的读操作不会乱序到屏障前的读操作，只针对读操作，不影响写操作。
```txt
store--------------------------+
load----------+                |
              |                |   ^
              v                |   |
------------smp_rmb()----------|---|------
              ^                |   |
              |                v   |
load----------+                    |
store------------------------------+

//备注:上述store箭头穿插,意思是可以插队执行,
//     即下面的store可以插队上面的store执行。
```

【回答】总结：写任何一个通用的程序我们都应该假设CPU类型是RMO模型。因为你不知道什么时候这段代码就可能在RMO模型上运行。内存屏障的使用应该成对，例如 smp_wmb 必须配对 smp_rmb 或者 smp_mb ，单独使用 smp_wmb 是达不到顺序效果的。同样 smp_rmb 必须配对使用 smp_wmb 或者 smp_mb 。



# 内存一致性模型TSO：WR乱序

# store buffer

【回答】TSO (Total Store Order)\
【回答】为了进一步加快内存访问速度，硬件引入了新的缓存 —— store buffer。

![[ad84da0f287f6b93347fe56b6f994cdb2e4caf9ee4cb509c3fbbff7d6507092c.png]]  

【回答】硬件工程师为了追求极致的性能，在CPU和L1 Cache之间又加入一级缓存，我们称之为store buffer。store buffer和L1 Cache还有点区别，store buffer只缓存CPU的写操作。store buffer访问一般只需要1个指令周期，这在一定程度上降低了内存写延迟。不管cache是否命中，CPU都是将数据写入store buffer。store buffer负责后续以FIFO次序写入L1 Cache。store buffer大小一般只有几十个字节。这里有个注意点，虽然store buffer主要是用来缓存CPU的写操作，但是CPU读取数据时也会检查私有store buffer是否命中，如果没有命中才会查找L1 Cache。这主要是为了CPU自己看到自己写进store buffer的值。

# store-load乱序
【回答】我们了解到每个CPU都会有自己私有L1 Cache。cache一致性和内存一致性有什么区别呢？一个关注的是一个地址，一个关注的是多个地址的顺序。由于 store buffer 的存在，CPU 对一个地址数据的写操作，其他 CPU 并**不能及时看见**。由于 CPU0 的执行顺序应该是 store-load ，在别的CPU看来乱序执行了，变成了load-store次序。这种内存模型，我们称之为完全存储定序(Total Store Order)。我们常见的PC处理器x86-64就是常见的TSO模型。

```cpp
static int x = 0, y = 0;
static int r1, r2;
static void int thread_cpu0(void)
{
  x = 1;    /* 1) */  //store
  r1 = y;   /* 2) */  //load

  //由于 store buffer 只缓存当前 CPU 的写操作，
  //其他 CPU 不能及时看到 x=1 这个结果，
  //因此其他 CPU 在读取 x 的值时，可能会读到旧值。
  //所以其他 CPU 可能会先看到2）后看到1）。
  //所以其他 CPU 可能会 load-store 的顺序，其实正确的顺序是 store-load
}
static void int thread_cpu1(void)
{
  y = 1;    /* 3) */
  r2 = x;   /* 4) */
}
static void check_after_assign(void)
{
  printk("r1 = %d, r2 = %d\n", r1, r2);
}
```

# 乱序解决办法
【回答】store 和 load 的组合有4种。分别是 store-store 、store-load 、load-load和load-store。**TSO模型中，只存在store-load存在乱序**，另外3种内存操作不存在乱序。\
【回答】硬件提供了内存屏障指令保证顺序一致性。Linux 内核中提供了 smp_mb() 宏对不同架构的指令进行封装，**smp_mb**() 的作用是阻止它后面的读写操作不会乱序到宏前面的指令前执行。他就像是个屏障一样，不容逾越。

```txt
store---------------------+
load----------+           |
              |           |
              v           v
-----------------smp_mb()--------------
              ^           ^
              |           |
load----------+           |
store---------------------+
```




# PERCPU

Basic concept: PerCPU technology generates a copy of a variable for each CPU, and each processor uses its own copy, thus avoiding mutual exclusion and synchronization between processors and improving the execution speed of the program.

Take the following actual variable as an example, the following variable is used to obtain the ID of the current CPU to determine which core is currently running on:
```c
#define smp_processor_id() __smp_processor_id()
#define __smp_processor_id() __this_cpu_read(cpu_number)
```

```c
#ifdef CONFIG_SMP
#define DEFINE_PER_CPU(type, name)                  \                               
    __percpu __attribute__((section(".data..percpu"))) type name;  \ 
#else   \
    __percpu __attribute__((section(".data"))) type name; \
#endif
```

Original link: https://zhuanlan.zhihu.com/p/363970087

In a single-core system, the percpu variable is placed in the `.data` section, so the percpu variable in the single-core system is directly treated as a normal variable.

In the SMP architecture, each variable of type `percpu` is placed in a specific `section` during compilation. For example, `.data..percpu` specified in the Linux kernel. After that, the kernel program will initialize a special memory area at runtime, and copy all the variables in this section, according to the number of CPU cores, copy multiple copies, each core has a copy, and each core can access its own set of copies. Although each CPU core accesses the same variable name, it only accesses its own address, and there is no access conflict, so there is no need for protection mechanisms such as mutual exclusion.

For example, the variable `cpu_number` in the Linux kernel is the Percpu type, which represents the ID of each CPU core.



# 原子访问：cmpxchg原理
---

以下是根据各大实现代码简化后的cmpxchg伪代码（Pseudocode）：
```cpp
int wei3dai4ma3_cmpxchg(void *ptr, unsigned long old, unsigned long new)
{
  if (*ptr == old)
  {
    *ptr = new;
    return old;
  } else {
    return *ptr;
  }
}
```

直接看上面伪代码也不容易看出其实际用途，因为cmpxchg最多用在锁上，所以我们就以锁相关的例子来辅助理解上述代码功能。必须从感知上理解了cmpxchg，才能更合理使用它。
```cpp
//把上述伪代码一些参数名称改变一下,来更精确表示其意思
ptr --> 锁当前状态
old --> 申请者期望状态
new --> 申请者占用锁时给锁设置的状态
```

下面以火车上两个互不相识的乘客A,B来访问同一个卫生间的场景来理解cmpxchg原理，其中乘客A先来到卫生间，在此之前卫生间没人。伪代码如下：
```cpp
//火车某车厢公共卫生间的锁 lock_state_cur
lock_state_cur == UNLOCK;//表示此时卫生间没人
lock_state_cur == LOCK;//表示此时卫生间有人


//此时过来一个乘客A想要使用卫生间
lock_state_cur == UNLOCK;//表示此时卫生间没人
lock_state_exp == UNLOCK;//乘客A希望卫生间没人
lock_state_use == LOCK;//乘客A正在使用卫生间时希望外面人知道卫生间此时有人

//cmpxchg会先上锁,然后返回UNLOCK,以告知乘客A可以使用卫生间.
cmpxchg(cur, exp, use) returns UNLOCK;
```

此时乘客B也来了，发现卫生间有人了，即卫生间显示上锁了(`lock_state_cur==LOCK`)，那么乘客B只能在卫生间外等待，伪代码如下：
```cpp
lock_state_cur == LOCK;//此时卫生间有人,乘客A在里面
lock_state_exp == UNLOCK;//乘客B希望卫生间没人
lock_state_use == LOCK;//乘客B正在使用卫生间时希望外面人知道卫生间此时有人

//cmpxchg发现锁当前状态为上锁状态,则会直接返回LOCK,以告知乘客B此时卫生间有人.
cmpxchg(cur, exp, use) returns LOCK;//LOCK表明卫生间有人,乘客B只能等待
```

乘客B等待的操作是，时不时瞟一眼卫生间，如果卫生间开锁了，那么乘客B就能进入卫生间，其他乘客以此类推。

cmpxchg的原理是，使用者只需要传递状态即可。如果锁当前状态为LOCK状态，则直接返回LOCK状态，以告诉申请者当前是锁住的状态。如果锁当前状态为UNLOCK状态，则表示资源可访问，此时申请者会把锁状态改变为LOCK状态，以防止其他申请者申请。

---

下面是基于ARM Cortex-A9 实现的原子 cmpxchg ：
```cpp
//whf@arrowpoint.com.cn
typedef struct {
  int counter;
} atomic_t;

//1=UNLOCK,0=LOCK
enum {
  ATOMIC_LOCK   = 0, //0=LOCK,caller can not access shared region
  ATOMIC_UNLOCK = 1  //1=UNLOCK,caller can access shared region
};

static atomic_t g_lock_access_ok = { ATOMIC_UNLOCK };//1=UNLOCK,0=LOCK
static inline int atomic_cmpxchg(int old, int new)
{
 unsigned long oldval, res;
 atomic_t *ptr = &g_lock_access_ok;

  //  smp_mb();
  dsb();

  do {
    __asm__ __volatile__("@ atomic_cmpxchg\n"
    "ldrex %1, [%3]\n"
    "mov %0, #0\n"
    "teq %1, %4\n"
    "strexeq %0, %5, [%3]\n"
    : "=&r" (res), "=&r" (oldval), "+Qo" (ptr->counter)
    : "r" (&ptr->counter), "Ir" (old), "r" (new)
    : "cc");
  } while (res);

  //  smp_mb();
  dsb();

 return oldval;
}
```

# Linux中断上下文

在进程上下文中运行的内核代码是可抢占的（preemptible），但中断上下文运行到最后，不能被抢占。 内核在进入中断之前关闭进程调度。 如果处理程序的类型为 IRQF_DISABLED，它会在执行时禁用所有本地中断。 不允许访问用户空间中的虚拟地址。




# Linux内核抢占计数器

  
抢占计数器其实就是一个特殊整型变量，下面是各个位域核心代码：
```cpp
#ifndef LINUX_PREEMPT_MASK_H
#define LINUX_PREEMPT_MASK_H
#include <linux/preempt.h>

/*
 * We put the hardirq and softirq counter into the preemption
 * counter. The bitmask has the following meaning:
 *
 * - bits 0-7 are the preemption count (max preemption depth: 256)
 * - bits 8-15 are the softirq count (max # of softirqs: 256)
 *
 * The hardirq count could in theory be the same as the number of
 * interrupts in the system, but we run all interrupt handlers with
 * interrupts disabled, so we cannot have nesting interrupts. Though
 * there are a few palaeontologic drivers which reenable interrupts in
 * the handler, so we need more than one bit here.
 *
 * PREEMPT_MASK:    0x000000ff
 * SOFTIRQ_MASK:    0x0000ff00
 * HARDIRQ_MASK:    0x000f0000
 *     NMI_MASK:    0x00100000
 * PREEMPT_ACTIVE:  0x00200000
 */

#define PREEMPT_BITS    8
#define SOFTIRQ_BITS    8
#define HARDIRQ_BITS    4
#define NMI_BITS    1

#define hardirq_count() (preempt_count() & HARDIRQ_MASK)
#define softirq_count() (preempt_count() & SOFTIRQ_MASK)
#define irq_count() (preempt_count() & (HARDIRQ_MASK | SOFTIRQ_MASK \
                 | NMI_MASK))

/*
 * Are we doing bottom half or hardware interrupt processing?
 * Are we in a softirq context? Interrupt context?
 * in_softirq - Are we currently processing softirq or have bh disabled?
 * in_serving_softirq - Are we currently processing softirq?
 */
#define in_irq()        (hardirq_count())
#define in_softirq()        (softirq_count())
#define in_interrupt()      (irq_count())
#define in_serving_softirq()    (softirq_count() & SOFTIRQ_OFFSET)

#endif /* LINUX_PREEMPT_MASK_H */
```

经典案例：计数器，网络接收数据包的程序可能先后运行在不同的 CPU 上，采用 percpu 变量进行计数，最后将所有 CPU 上的 percpu 副本相加就可以得到所接收数据包的总和。

```cpp
//但是这个返回的地址并非是可以直接使用的变量地址，就像静态定义的那样，
//这只是一个原始数据，
//真正被使用的数据被 copy 成 n(n=CPU数量) 份分别保存在每个 CPU 独占的地址空间中，
//在访问 percpu 变量时就是对每个副本进行访问。
type __percpu *ptr alloc_percpu(type);

//对于静态的变量读写:
//通过这个函数获取当前执行这段代码的 CPU 所对应的 percpu 变量
get_cpu_var(name) //关闭内核抢占
put_cpu_var(name) //表示当前操作的结束(使能内核抢占)
//事实上它仅仅是使能了在 get_cpu_var 函数中关闭的内核抢占。

//使用流程示例:
DEFINE_PER_CPU(int, val);
...
int *pint = &get_cpu_var(val)
*pint++;
put_cpu_var(val)
```

为什么在调用 get_cpu_var 时，第一步是禁止内核抢占呢? 想想这样一个场景，进程 A 在 CPU0 上执行，读取了 percpu 变量到寄存器中，这时候进程被高优先级进程抢占，当这段代码继续执行的时候可能被转移到 CPU1 上执行，这时候在 CPU1 执行的代码操作的仍旧是 CPU0 上的 percpu 变量，这显然是错误的。


```cpp
per_cpu_ptr(ptr, cpu);
//这个接口允许指定 CPU ，不再是只能获取当前 CPU 的值

//通常操作是这样的:
int *pint = alloc_percpu(int);
...
int *p = per_cpu_ptr(pint,raw_smp_processor_id());
(*p)++;
```

其中，raw_smp_processor_id() 函数返回当前 CPU num，这个示例也就是操作当前 CPU 的 percpu 变量，这个接口并不需要禁止内核抢占，因为不管进程被切换到哪个 CPU 上执行，它所操作的都是第二个参数提供的 CPU。

Declare a variable and store the variable in the specified section.

```cpp
#define get_cpu_var(var)						\
(*({									\
	preempt_disable();						\
	this_cpu_ptr(&var);						\
}))

#define put_cpu_var(var)						\
do {									\
	(void)&(var);							\
	preempt_enable();						\
} while (0)
```

# RCU消息发布与订阅

RCU全称Read-Copy-Update，是Linux内核互斥设施中一种免锁机制。前面的加锁解锁都涉及到内存操作，同时还伴随内存屏障，这些使得锁操作的系统开销很大。RCU主要应用在那种，读线程多，而写线程唯一的情况，同时需要保证数据一致性。

1，在读取过程中，另外一个线程删除了一个节点。删除线程可以把这个节点从链表中移除，但它不能直接销毁这个节点，必须等到所有的读取线程读取完成以后，才进行销毁操作。RCU中把这个过程称为宽限期（Grace period）。

RCU读取者的性能提升是在增加写入者负担的前提下完成的。因此如果写入者的操作比例在10%以上，就应该考虑其他互斥操作，不应该考虑RCU。RCU适用于大量读取者的互斥情况。
