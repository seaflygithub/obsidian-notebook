## 使用配置文件

- 方法1：通过命令行加载配置文件来格式化源代码
- 方法2：使用vscode的保存操作来触发对源代码的格式化

### 命令行方式
确保当前目录下或者祖先目录下已有 .clang-format 文件，然后直接：
```bash
cp -v .clang-format ~/
clang-format -i test.c
```

### vscode方式
在打开的工程顶层目录下新建 .vscode 目录，并在 .vscode 目录下新建 .clang-format 文件，配置文件内容参考前面的配置文件内容，保存后即可使用。编辑源代码，在vscode编辑源代码并执行保存时，会自动格式化你当前编辑的源代码。

## 一种简单的qemu网络配置方法

一种简单的qemu网络配置方法：https://blog.csdn.net/wujianyongw4/article/details/80497528
qemu上网是一件头疼的事，搜索了很多方法都是不可靠的，最终想到了一种权宜之计。qemu上网的一种主要的方式是在host上配置好网桥，将tap设备添加到网桥上，在起qemu虚拟机的时候用tap作为网络设备，这种方法的关键是怎么在host上配置可以上网的网桥。利用现成的可以用的网桥就是我要讲的权宜之计，docker0就是这种可以用的网桥。docker默认的网络模式是桥接，就是在host上创建一个docker0，容器通过docker0实现上网。虚拟机也可以通过docker0上网。下面说下一下具体怎么做。

```bash
#首先得有docker
sudo apt-get install -y docker.io

#然后启动docker服务（启动之后，这时候用ifconfig就可以查看到docker0的存在）
sudo systemctl start docker
```

```bash
#安装tunctl命令支持
sudo apt-get install -y uml-utilities

#创建tap0(创建之后，这时候用ifconfig就可以查看到tap0的存在)
sudo tunctl -t tap0 && sudo ifconfig tap0 up

#将tap0挂到docker0上
sudo brctl addif docker0 tap0

#查看一下是不是挂上去了
brctl  show
```


使用qemu启动虚拟机时加上：`-net nic -net tap,ifname=tap0,script=no,downscript=no`

进入虚拟机给网卡配置网络，网段跟docker0一致，将路由配置成docker0的ip
```bash
ifconfig eth0 172.17.0.222/24
route add default gw 172.17.0.1
```

配置qemu内虚拟机DNS服务：`echo "nameserver 8.8.8.8" > /etc/resolv.conf`
（这时你就可以在虚拟机中上网了）

如果你要在qemu虚拟机中跑docker那么你就得稍微修改一下，在host上起docker之前先在/etc/docker/daemon.json中改一下默认的ip，例如：`"bip": "192.168.10.1/24"`。这样在启动docker之后，docker0的ip就变成了"192.168.10.1",这样就不会跟qemu虚拟机里的docker有冲突了。


```bash
sudo service docker restart
sudo tunctl -t tap0 && sudo ifconfig tap0 up
sudo brctl addif docker0 tap0
echo "-net nic -net tap,ifname=tap0,script=no,downscript=no (for qemu argument)"
```

## imx6ull激活交叉编译环境设置

- 设置交叉环境变量：setenvironment.sh
```bash
#!/bin/bash
# 脚本作用：根据用户输入找到匹配envname，然后设置对应的PATH信息。

# 支持的环境名称，也就是支持的交叉工具链清单(一一对应)
envnames=("songbaohua" "imx6ull" "zhuomufiberimage")
envpaths=(
		"/home/linux/${envnames[0]}/gcc-arm-none-eabi/bin"
		"/home/linux/${envnames[1]}/gcc-linaro-4.9.4-2017.01-x86_64_arm-linux-gnueabihf/bin"
		"/home/linux/${envnames[2]}/ext-toolchain/bin"
		)
# 原始的PATH环境变量值
origpath=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin:/usr/games:/usr/local/games:/snap/bin

function show_supported_envnames()
{
	for ((i=0;i<${#envnames[*]};i++))
	do
		envname=${envnames[i]}
		echo "Now supported envname: ${envname}"
		echo "              envpath: ${envpaths[i]}"
	done
}

if [ $# -lt 1 ] ; then 
	echo "  usage:source setenvironment.sh <envname>"
	# 遍历输出支持的环境名称数组成员

	show_supported_envnames
	return 1
fi
inputname="${1}"
findout=0

# 初始化默认PATH环境变量值
export PATH=${origpath}

# 根据支持的环境名称来追加新的环境变量值
for ((i=0;i<${#envnames[*]};i++))
do
	envname=${envnames[i]}
	if [ "${envname}" = "${inputname}" ];then
		findout=1
		echo "setting ${inputname} workpath:${envpaths[i]} ..."
		export PATH=${PATH}:${envpaths[i]}
		echo "setting ${inputname} workpath:${envpaths[i]} successfully!"
	fi
done

if [ ${findout} -ne 1 ]; then
	# 遍历了整个数组后，没有找到支持的环境名称，无法设置环境路径信息
	echo "Not found supported \"${inputname}\" in \"envnames\" array"

	show_supported_envnames
fi
```

## 6.1——ftrace

（function trace）基于静态代码插桩技术，利用 gcc 编译器的 profile 特性在所有函数入口添加了一段插桩(stub)代码， ftrace 重载这段代码来实现 trace 功能。gcc 编译器的 "-pg" 选项会在每个函数入口处加入 mcount 的调用代码，原本 mcount 有 libc 实现，因为内核不会链接 libc 库，因此 ftrace 编写了自己的 mcount stub 函数。当然，内核配置也需要使能这些功能：
```txt
CONFIG_FTRACE=y
CONFIG_HAVE_FUNCTION_TRACER=Y
CONFIG_HAVE_FUNCTION_GRAPH_TRACER=Y
CONFIG_HAVE_DYNAMIC_FTRACE=y
CONFIG_FUNCTION_TRACER=y
CONFIG_IRQSOFF_TRACER=y
CONFIG_SCHED_TRACER=y
CONFIG_ENABLE_DEFAULT_TRACERS=y
CONFIG_FTRACE_SYSCALLS=y
CONFIG_PREEMPT_TRACER=y
```

ftrace 通过 debugfs 文件系统向用户空间提供访问接口，因此需要在系统启动时挂在 debugfs ，可以修改系统的 /etc/fstab 文件或手工挂在。
```bash
mount -t debugfs debugfs /sys/kernel/debug
```

在 /sys/kernel/debug/trace 目录下提供了各种跟踪器（tracer）和 event 事件，一些常用选项如下：
- available_tracers:列出当前系统支持的跟踪器。
- available_events: 列出当前系统支持的event事件。
- current_tracer : 可通过echo/cat设置和显示当前正在使用的跟踪器。默认为 nop ，不做任何跟踪。
- trace : 读取跟踪信息。通过 cat 命令查看 ftrace 记录下来的信息。
- tracing_on : 用于开始或暂停跟踪。
- trace_options : 设置 ftrace 的一些相关选项。

ftrace 当前包含多个跟踪器，很方便用户用来跟踪不同类型的信息，例如进程睡眠唤醒
抢占延迟的信息。查看 available_tracers 可以知道当前系统支持哪些跟踪器，如果没有用户想要的，就必须在配置内核时自行打开，然后重新编译内核。
- nop ： 不跟踪任何信息。将 nop 写入 current_tracer 可清空。
- function : 跟踪内核函数执行情况。
- function_graph : 显示类似C语言的函数调用关系图。
- wakeup : 跟踪进程唤醒信息。
- irqsoff : 跟踪关闭中断信息，并记录关闭的最大时长。
- preemptoff : 跟踪关闭禁止抢占信息，并记录关闭的最大时长。
- preemptirqsoff : 结合上述两个功能。
- sched_switch : 对内核中的进程调度活动进行跟踪。

## 6.5——死锁检测

死锁（deadlock）是指两个或多个进程因争夺资源而造成的互相等待的现象。在内核开发中，时常需要考虑并发设计，即使采用正确的编程思路，也不可避免发生死锁。
- 递归死锁：例如在中断等延迟操作中使用了锁，和外面的锁构成了递归死锁。
- AB-BA死锁：多个锁因处理不当而引发死锁。

```cpp
#include <linux/init.h>
#include <linux/module.h>
#include <linux/kernel.h>
static DEFINE_SPINLOCK(hack_spinA);
static DEFINE_SPINLOCK(hack_spinB);
void hack_spinAB(void)
{
	printk("hack_ lockdep: A->B\n");
	spin_lock(&hack_spinA);
	spin_lock(&hack_spinB);
}
void hack_spinBA(void)
{
	printk("hack_lockdep: B->A\n");
	spin_1ock(&hack_spinB);
}
static int lockdep_test_init(void)
{
	printk("figo: my lockdep module init\n");
	hack_spinAB();
	hack_spinBA();
	return 0;
}
```

因为 hack_spinB 已经被成功获取并且还没有释放，它会一直等待，而且它也被锁在 hack_spinA 的临界区里。

## Linux驱动错误日志导出方法（用户空间监测）

- dmesg打印信息如何后台保存到文件：https://blog.csdn.net/bingqingsuimeng/article/details/13769023
- dmesg调试信息常常作为判断系统异常退出的重要信息，但是当系统异常退出或重启时，信息又难以保存下来，于是才有了将dmesg调试信息保存到文件中的想法（capdmesg.c）：
```cpp
#include <stdio.h>
#include <unistd.h>
#include <stdlib.h>
#include <string.h>
#include <errno.h>
#include <time.h>
#include <sys/klog.h>
#include <sys/stat.h>
#include <sys/types.h>
#include <unistd.h>

#define BUFFER_LEN_MSG (256 * 1024) //256KB

int main(int argc, char *argv[])
{
	char msgbuf[BUFFER_LEN_MSG] = {0};
	char tmpbuf[BUFFER_LEN_MSG] = {0};

	int ret = 0;
	FILE *fp = NULL;
	FILE *fp_log = NULL;
	struct tm *ptr;
	time_t lt;
	int errcnt = 0;

	//check log file argument
	if (argc != 2)
	{
		fprintf(stderr,"usage:%s <logfile>\r\n", argv[0]);
		return -1;
	}
	const char *logPathFileName = argv[1];
	
	//check user permission
	uid_t uid = getuid();
	if (uid != 0) //root's uid == 0
	{
		fprintf(stderr,"Error:Need root permission\r\n");
		return -1;
	}

	//loop for watching dmesg kernel info refresh
	while(1)
	{
		fp = fopen(logPathFileName, "a+");

		if(fp != NULL)
		{
			//note:need root permission
			ret = klogctl(4, msgbuf, BUFFER_LEN_MSG);
			if(ret > 0)
			{
				//generate date time information
				lt = time(NULL);
				ptr = (struct tm *)localtime(&lt);
				sprintf(tmpbuf, "[%s", asctime(ptr));
				tmpbuf[strnlen(tmpbuf, BUFFER_LEN_MSG) - 1] = ']';
				tmpbuf[strnlen(tmpbuf, BUFFER_LEN_MSG) - 0] = '\0';

				//combine date time information and kernel message
				strncat(tmpbuf, msgbuf, strnlen(msgbuf, BUFFER_LEN_MSG));

				//write combined information into log file
				fwrite(tmpbuf, 1, strlen(tmpbuf),fp);
				fflush(fp); fclose(fp); fp = NULL;

				memset(tmpbuf, 0, BUFFER_LEN_MSG);
				memset(msgbuf, 0, BUFFER_LEN_MSG);
			}
			else
			{
				fclose(fp); fp = NULL;
			}
		}
		else
		{
			fp_log = fopen("/dmesg_daemon_error.log", "a+");
			if (fp_log != NULL)
			{
				fprintf(fp_log, "Error:fopen %s error (reason:%s)\r\n", 
					logPathFileName, strerror(errno));
				fclose(fp_log); fp_log = NULL;
			}
			fprintf(stderr, "Error:fopen %s error (reason:%s)\r\n", 
				logPathFileName, strerror(errno));
			errcnt++;
			if (errcnt >= 512)
			{
				return -1;
			}
		}
	}
	return 0;
}
```

- 编译方法（Makefile）：```$(CC) $(TARGET).c -o $(TARGET).elf -Wall```
- 运行：```sudo dmesg-daemon.elf dmesg_issue.log &```
- 程序就进入后台执行了，开始死循环检测是否有新信息输出，一旦有新信息就立刻输出到指定的日志文件里。（该方法就是轮询，CPU占用率非常高，后续抽空优化或换其他方法）

# 关于死锁的调试

## 解决死锁的基本方法（预防死锁）

- 一次性分配所有资源，这样就不会再有请求了（破坏请求条件）
- 只要有一个资源得不到分配，也不给这个进程分配其他的资源（破坏请保持条件）
- 即当某进程获得了部分资源，但得不到其它资源，则释放已占有的资源（破坏不可剥夺条件）
- 系统给每类资源赋予一个编号，每一个进程按编号递增的顺序请求资源，释放则相反（破坏环路等待条件）

### 预防办法：以确定的顺序获得锁
- 线程A：锁住a ---> 尝试锁住b ---> 永久等待
- 线程B：锁住b ---> 尝试锁住a ---> 永久等待
- 预防办法（以确定的顺序获得锁）：
  - 线程A：锁住a ---> 尝试锁住b ---> 永久等待
  - 线程B：锁住a ---> 尝试锁住b ---> 永久等待

### 预防办法：超时放弃

【回答】在编写驱动或者内核模块时，常常会显式或隐藏地对指针进行非法取值或者使用不正确的指针，导致内核发生一个 oops 错误。当处理器在内核空间访问一个非法的指针时，因为虚拟地址到物理地址的映射关系没有建立，触发一个缺页中断，在缺页中断中因为该地址是非法的，内核无法正确地为该地址建立映射关系，因此内核触发了一个 oops 错误。
